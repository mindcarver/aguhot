# Epic 1 Context: 可信热点发布闭环

<!-- Generated from planning artifacts. Regenerate with compile-epic-context if planning docs change. -->

## Goal

Establish the closed loop that lets users see a publicly displayable hot-event stream, open a detail page to read the summary and evidence timeline, with everything that reaches the public surface passing through a minimal operator review and publish gate. The epic delivers "viewable" and "trustworthy" together — a candidate event is never shown publicly until it clears the publish workflow.

## Stories

- Story 1.1: 初始化公共站点脚手架与匿名首页壳层
- Story 1.2: 基础导航与响应式公共页面壳层
- Story 1.3: 视觉 token 与排版基础落地
- Story 1.4: 证据源采集与归档
- Story 1.5: 候选热点聚类与待复核生成
- Story 1.6: 运营复核队列与发布闸门
- Story 1.7: 公开热点事件流
- Story 1.8: 热点事件详情、证据时间线与解释分区
- Story 1.9: 已发布热点的文案与标签修正
- Story 1.10: 已发布热点的合并、拆分与下线发布

## Requirements & Constraints

- The homepage must show a priority-sorted hot-event stream, not a raw article list; each card shows title, one-line explanation, source count, latest update time, and a human-readable ranking reason (multi-source coverage, recent heating, or clear market reaction).
- Users can filter the stream by at least one date dimension and one category dimension; active filters are always visible and clearable without losing reading context.
- The detail page must present three distinct first-screen blocks: "what happened", "why it matters", "what remains uncertain" — confirmed facts, system explanation, and unconfirmed judgment must be visually separated, never blended into one info blob.
- The evidence timeline lists each source with name, time, summary, and original link, default chronological; if an original link is dead the record stays with a clearly marked status rather than silently disappearing.
- Every publicly shown hot event must trace back to at least one valid evidence source (100% traceability coverage).
- All AI-generated summaries, explanations, and derived content must carry a uniform AI label, identical on public and operator pages.
- Target ingest-to-public-stream latency is under 10 minutes for in-scope priority sources.
- Operator review is the publish gate: events move through `ReviewDecision` and `PublicationDecision`; public visibility is controlled solely by `publication_status`. Candidates not yet published never appear on public pages.
- Operators can merge, split, revise title/labels/explanation, take offline, and re-publish already-published events; every such action is an append-only version or decision — never an in-place overwrite of history. Public pages only show the current published version.
- Operator review on explanation quality and source completeness; operator can mark an event as pending-review, reviewed, or needs-takedown, and can block publication when sources are insufficient or explanation is clearly unreliable.
- Audit records are mandatory for all review-related key changes.
- Graceful degradation: if external data (e.g., market reaction signals) is unavailable, the product still shows hot events and the evidence timeline with explicit missing-status notices. Empty states never render placeholder or fake data.
- The product must not imply investment advice; no buy/sell/target-price/position wording in any page, label, or CTA.

## Technical Decisions

- **Paradigm:** modular monolith with event-driven ingest pipeline. Two runtimes — `apps/web` (request/response only) and `apps/worker` (ingest, normalize, cluster, explain, publish). No microservices in V1.
- **Single write-owner per aggregate:** `source-ingest` owns raw `EvidenceSource`; `event-assembly` owns `HotEvent` clustering and merge/split; `review-workflow` owns `ReviewDecision`/`PublicationDecision`. Modules never write across each other's boundaries; cross-aggregate change goes via application commands or queue events.
- **Public reads only published read models:** all public pages/APIs read `published_*` materialized read models generated by `publish-orchestrator`, never raw ingest or operator working tables.
- **Heavy work is async:** ingest, extraction, dedup, clustering, explanation generation, publish-model refresh all run as BullMQ jobs. The web request path must not synchronously await scraping or LLM calls.
- **Versioned, append-only records:** `ExplanationVersion`, `ReviewDecision`, `PublicationDecision` are append-only. Public shows current `published`; operator console sees full version/source/status chains.
- **Adapter ports for all externals:** `SourceAdapter`, `MarketDataAdapter`, `LLMAdapter`. Domain modules must not import third-party SDKs directly; provider swaps happen only in adapter/worker-assembly layers.
- **Identity is not a public-path dependency:** public browsing, search, detail, and timeline reading are anonymously usable; `user-profile` only owns follow-list and preferences and must not block public content.
- **Stack (fixed baseline):** Node.js 24.18 LTS, TypeScript 5.9, Next.js 16 App Router, React 19.2, Tailwind 4, shadcn/ui 4, PostgreSQL 18, Prisma 7.7, Redis 8, BullMQ 5.79, Playwright 1.60.
- **Structural seed:** `apps/web` split into `(public)` and `(operator)` route groups; `apps/worker` holds `queues/` and `jobs/` (ingest / cluster / explain / publish / digest); `packages/core` holds domain modules (`source-ingest`, `event-assembly`, `publish-orchestrator`, `review-workflow`) plus shared `contracts/` (command/query DTOs, ports) and `shared/` (ids, time, errors, tracing).
- **Conventions:** primary keys are UUIDv7; times stored UTC, surfaced as ISO 8601; public API uses `data / meta / error` envelope; every request and job carries a `trace_id`; domain entities are singular PascalCase, DB tables snake_case plural, queue/job names kebab-case, domain events past-tense PascalCase (e.g., `HotEventPublished`); decimals stored raw and formatted at the display layer.
- **Deferred (not decided in V1):** specific cloud/deployment platform, specific data-source procurement list, whether to introduce a dedicated full-text search engine (PostgreSQL capabilities are acceptable for now), whether low-risk events can auto-publish (the review gate stays mandatory until real operator load is observed).

## UX & Interaction Patterns

- **Information architecture:** desktop uses a left-rail primary nav (首页 / 日报 / 主题 / 收藏 + internal entry); mobile collapses to top nav + drawer; nav depth is one level; any detail page must return to its originating consumption context.
- **Responsive breakpoints:** `≥1200px` left nav fixed; `768–1199px` nav narrows/collapses, filters float to top; `<768px` drawer nav, cards stack vertically. Mobile is not a compressed screenshot — it must preserve hotspot scanning, readable evidence timeline, and clear return paths.
- **Visual system (DESIGN.md tokens):** warm canvas + deep-ink text; brand color and market red/green are fully decoupled (red/green serve only market semantics, never brand); Source Han Serif SC for display titles, Source Han Sans SC for body/headline, IBM Plex Mono for numerics. AI label uses accent-warm only.
- **Component behaviors for this epic:** event card is whole-card clickable to detail; filter pill has clear default/active/clear states; evidence row reads like an editor footnote with a brand-color left rule; AI label is consistent across all AI-generated content; operator review-status marker distinguishes pending / reviewed / needs-takedown and binds public visibility.
- **Trust & provenance:** every event explicitly shows source count, latest update, and a traceable source entry; explanation and fact content stay visually partitioned; uncertain state is allowed to remain visible rather than forced into a confident conclusion; degraded/missing data shows explicit status, not silent hiding.
- **State patterns:** cold start shows skeleton near real list density (no fake data); no-hotspots state shows "currently no publicly displayable hot events" with last-update time; insufficient evidence keeps the summary but marks "sources insufficient / still pending"; dead links keep time and source but mark link inaccessible; taken-down event shows "withdrawn or paused" publicly while the operator side retains the reason.
- **Interaction primitives:** homepage scan-first (first screen answers "what is the market trading today" without requiring a click); scroll is the default reading mode — no horizontal carousel for core hotspots; overlays only for short tasks (filters, login prompt), never for core explanation or timeline; banned patterns include auto-play, disruptive animation, multi-layer popups, and color-only status semantics.
- **Accessibility floor:** keyboard-reachable for cards, search, filters, source links; structured heading hierarchy; market semantics must carry text/symbol labels in addition to color; mobile tap targets meet basic touch sizes; reduced-motion preference triggers instant state changes.

## Cross-Story Dependencies

- Stories 1.1–1.3 (scaffold, nav shell, design tokens) must land before any public surface (1.7, 1.8) — they are the shared web baseline.
- Stories 1.4 (ingest/archive) → 1.5 (clustering/candidate generation) → 1.6 (review queue/publish gate) form the worker-side pipeline; 1.7 (public stream) and 1.8 (detail/timeline) can only read what 1.6 publishes.
- Stories 1.9 (copy/label revision) and 1.10 (merge/split/takedown/re-publish) both depend on 1.6's versioned `ReviewDecision`/`PublicationDecision` records and the `publish-orchestrator` read-model refresh — they reuse, not rebuild, the publish gate.
- The public read contract (published read models) is shared by 1.7 and 1.8; changes to the publish-state read model ripple across both.
- Upstream dependency on Epic 2 is minimal for publish-loop functionality — market-reaction and theme-linking modules are not required for Epic 1 to demonstrate "viewable + trustworthy"; but the published read model contract established here is the integration point Epic 2 and Epic 3 will consume.
