## 1. Vision

AGUHOT 是一个 A 股优先的市场动态时间流与 AI 解释产品。它不是交易终端，也不是投顾工具，而是把分散在公告、新闻、快讯、研报和市场异动中的线索，以分钟级 `时间流` 形态呈现在首页，并对每条动态附以 AI 生成的 `AI 解读`，对关键事件提供 `AI 深读` 与跨事件 `趋势研判`，帮助用户快速判断”现在市场正在交易什么、为什么重要”。

产品的核心价值是”让用户用最低成本跟上市场节奏，且每条动态都有可追溯证据与 AI 解释背书”。用户打开产品后，先在 `时间流` 中按时间倒序扫描当日动态与 `AI 解读`，点开任一条进入 `热点事件详情页`，沿 `证据时间线` 看清这件事从哪里来、为什么重要、哪些 `市场反应信号` 已经出现、哪些地方仍然不确定。

首页 `时间流` 顶部保留一条”今日重点/市场主线”置顶带（top-N saliency，常态启用），确保首页仍主动回答”市场正在交易什么”。产品的最终判断标准不是”用户刷了多少条时间流”，而是”用户离开产品时是否更清楚市场正在交易什么”——时间流与三层 AI 分析都是达成该判断的形态，不是目的本身。

[ASSUMPTION: V1 以公开 Web 产品为首发形态，优先验证“热点发现 + 热点解释”的产品价值，不在第一阶段扩展到交易、社区或机构终端。]

### 1.1 Why Now

现在做这件事成立，原因有三点：

- A 股信息供给已经极度充沛，但信息组织方式仍然以来源为主，不以事件为主。
- 现有强产品要么偏资讯流量，要么偏专业终端，留出了“轻研究、强解释”的中间层空间。
- 大模型降低了摘要、归纳、证据整理和日报生成成本，使“解释层产品”第一次有机会跑到足够高的频率和足够低的成本。

## 2. Target User

### 2.1 Jobs To Be Done

- 在开盘前或盘中快速判断今天最值得关注的市场主线。
- 在出现异动时，快速找到最早、最核心的 `证据源`，而不是在多个 App 和媒体间来回切换。
- 把“这条消息”和“哪些板块/个股在响应”连起来看，而不是分散消费。
- 在收盘后快速复盘当天热点，整理成可复用的判断材料。
- 用更低成本完成轻量投研、内容选题或市场跟踪。

### 2.2 Non-Users (v1)

- 需要完整机构级数据库、模型库、Excel 插件和终端工作台的专业机构研究员。
- 期待直接获得买卖建议、目标价、仓位建议或自动策略信号的用户。
- 主要关注美股、港股、加密货币或全球宏观交易的用户。

### 2.3 Key User Journeys

- **UJ-1. 陈明在开盘前 10 分钟快速扫主线。**
  - **Persona + context:** 陈明是高频关注 A 股的个人投资者，每天开盘前会看公告、财经媒体和社交平台，但没有机构终端。
  - **Entry state:** 已打开 AGUHOT 首页，默认进入当日 `热点事件流`。
  - **Path:** 浏览前 5 个 `热点事件` -> 查看每个事件的一句话解释 -> 点开最热事件的 `证据时间线` -> 查看受影响 `概念`、`行业`、`个股` 与 `市场反应信号`。
  - **Climax:** 陈明在 3 分钟内知道今天最强主线是什么，为什么是它，以及市场是否已经开始反应。
  - **Resolution:** 陈明带着明确优先级离开首页，继续跟踪其中 1-2 个 `热点事件`。

- **UJ-2. 林瑶在盘中追溯一条突然拉升的题材。**
  - **Persona + context:** 林瑶是内容创作者兼重度行情用户，看到一个板块异动后需要快速解释原因。
  - **Entry state:** 从搜索或相关入口进入某个 `热点事件详情页`。
  - **Path:** 查看事件摘要 -> 按时间阅读 `证据时间线` -> 对照 `市场反应信号` -> 打开原始链接确认表述 -> 收藏该 `热点事件` 以便稍后写作。
  - **Climax:** 林瑶能确认这次异动背后的主要触发因素，而不是只看到结果。
  - **Resolution:** 她保存了可追溯材料，能继续做内容输出或进一步判断。

- **UJ-3. 阿周在收盘后快速复盘全天热点。**
  - **Persona + context:** 阿周是轻研究用户，希望用 10 分钟内完成当日热点复盘。
  - **Entry state:** 进入当日 `日报` 页面。
  - **Path:** 阅读今日热点摘要 -> 逐条查看代表性 `热点事件` -> 跳转到持续发酵的 `主题页` -> 对比同主题近几日事件变化。
  - **Climax:** 阿周完成一份结构化复盘，不需要自己重新拼接当天资讯。
  - **Resolution:** 他对哪些 `主题页` 需要第二天继续关注形成明确判断。

## 3. Glossary

- **热点事件** — 被系统归并后的单个市场热点单元，由多个 `证据源` 和一条事件级解释构成。
- **证据源** — 支撑 `热点事件` 的原始材料，如公告、新闻、快讯、研报摘要或权威公开链接。
- **证据时间线** — 按时间顺序展示某个 `热点事件` 所对应的 `证据源` 列表。
- **市场反应信号** — 用来描述市场对某个 `热点事件` 已有反应的结构化信息，如板块涨跌、成交放量、涨停家数、异动表现。
- **主题页** — 围绕同一长期主题或连续主线聚合的页面，包含多个相关 `热点事件`。
- **日报** — 对一个交易日内重点 `热点事件` 的结构化总结页面。
- **关注列表** — 用户手动收藏的 `热点事件` 与 `主题页` 集合。
- **运营复核** — 对 `热点事件` 的归组、标题、解释、标签与来源进行人工检查和修正的内部流程。

## 4. Features

### 4.1 热点事件流

**Description:** `热点事件流` 是产品首页的默认核心视图，负责把零散资讯组织成可快速浏览的 `热点事件` 列表。每条卡片必须让用户在极短时间内理解“发生了什么、为什么现在重要、市场是否已经在反应”。实现 UJ-1。

**Functional Requirements:**

#### FR-1: 默认展示当日市场动态时间流

用户打开首页后直接看到按时间倒序排列的当日 `时间流`，每条动态对应一个 `热点事件` 或一条 `证据源` 条目，并附带 AI 生成的 `AI 解读`。实现 UJ-1。

**Consequences (testable):**
- 默认视图展示当天的 `时间流`，按分钟级时间倒序排列，按交易日/自然日分组。
- 同一 `热点事件` 的多条 `证据源` 折叠为"同事件精选"条目，展开可见各来源；未聚类的单源快讯独立成条。
- 每条动态至少展示：时间戳、来源名称、标题、一句话摘要、`AI 解读`（带 AI 标识）、`证据源` 数量。
- 用户可按时间范围与类别（概念 / 行业 / 个股 / 公告 / 研报）筛选，筛选条件可见且可清除（对齐 FR-2）。
- 时间流顶部以编号式「当前热点」排行呈现 top-N（ordered list，紧凑编号 1. 2. 3…，每项 = 标题 + 来源数 + 相对时间），主动回答”市场正在交易什么”。复用既有 saliency 读模型（`listPublishedHotEvents`），非新读模型/字段（2026-07-12 sprint-change-proposal：原 MainLineBand 卡片带 → 编号排行）。
- 视觉形态：`时间流` 以无边框编辑型纵栏呈现（UX-DR16），按交易日分节，每条以 HH:mm 领头、来源名提级、AI 解读实线签名块；桌面端用顶部窄条极简导航（UX-DR3），内容区居中窄栏。
- 当没有可展示的动态时，页面必须给出明确空状态说明与最近更新时间。

#### FR-2: 支持按时间和类别筛选热点事件流

用户可以按时间范围和类别筛选 `热点事件流`。实现 UJ-1。

**Consequences (testable):**
- 用户可以切换至少一个日期维度和一个类别维度。
- 筛选操作不会打断用户当前阅读上下文。
- 当前筛选条件在页面上可见且可清除。

#### FR-3: 展示同事件精选与置顶理由

用户可以看到某条 `时间流` 动态为何被”同事件精选”折叠或临时置顶。实现 UJ-1。

**Consequences (testable):**
- 系统至少向用户暴露”多源覆盖””近期升温”或”市场反应明显”等非公式化理由，说明该条目被精选/置顶的依据。
- 理由是可读文案，不要求暴露内部打分细节。
- 理由必须与该 `热点事件` 的实际数据状态一致，不得编造。
- `时间流` 的主排序仍为时间倒序；精选/置顶理由只在偏离时间序时出现。
- 顶部编号式「当前热点」排行复用既有 saliency 读模型（`listPublishedHotEvents`），不引入新读模型或精选分字段；不伪造「精选 NN」编辑质量分（NFR-2），以「来源数」chip 替代（2026-07-12 sprint-change-proposal 追加）。

### 4.2 热点事件详情与证据时间线

**Description:** `热点事件详情页` 是产品的价值中心，负责把首页的“看见热点”升级成“理解热点”。页面必须把解释、来源和不确定性放在一起，而不是只给二次加工后的结论。实现 UJ-1、UJ-2。

**Functional Requirements:**

#### FR-4: 提供事件级摘要与关键结论

用户进入 `热点事件详情页` 后可以快速看到该事件的摘要与关键结论。实现 UJ-1、UJ-2。

**Consequences (testable):**
- 页面首屏至少展示事件摘要、为什么重要、当前不确定点三个信息块。
- 摘要内容与 `证据时间线` 中的主要 `证据源` 一致，不得脱离来源独立编造。
- 摘要如果由 AI 生成，必须被标识。

#### FR-5: 提供可追溯的证据时间线

用户可以按时间阅读支撑该 `热点事件` 的 `证据时间线`。实现 UJ-1、UJ-2。

**Consequences (testable):**
- 每个 `证据源` 至少展示来源名称、时间、摘要和原始链接。
- `证据时间线` 默认按时间顺序展示，并允许用户快速跳回最早或最新证据。
- 当某个原始链接失效时，系统仍应保留来源记录并标记状态。

#### FR-6: 明确区分事实、解释和不确定性

用户在 `热点事件详情页` 中可以区分已确认事实、系统解释和待确认判断。实现 UJ-2。

**Consequences (testable):**
- 页面中的解释性内容不能伪装成原始事实。
- 系统必须提供至少一种方式表达“尚未确认”或“仍有争议”的状态。
- 人工或系统修订解释时，应保留当前版本对用户可见的一致性。

### 4.3 市场反应与关联展示

**Description:** 产品必须把 `热点事件` 和市场动作联动起来，否则只会退化成资讯整理工具。这个能力的重点不是给交易建议，而是帮助用户理解“消息”和“市场反应信号”之间的关系。实现 UJ-1、UJ-2。

**Functional Requirements:**

#### FR-7: 展示热点事件关联的概念、行业和个股

用户可以看到某个 `热点事件` 主要关联的 `概念`、`行业` 和代表性 `个股`。实现 UJ-1、UJ-2。

**Consequences (testable):**
- 详情页至少展示一组关联 `概念`、`行业` 或 `个股`。
- 关联项必须与当前 `热点事件` 有明确映射依据，不能完全手工随意填写。
- 关联项点击后有明确去向，如过滤结果页、详情页或二级视图。

#### FR-8: 展示热点事件对应的市场反应信号

用户可以查看该 `热点事件` 已出现的 `市场反应信号`。实现 UJ-1、UJ-2。

**Consequences (testable):**
- `市场反应信号` 至少包含一种价格或成交维度信号和一种板块或涨停维度信号。
- 每个信号都要有时间语境，避免与其他交易时段混淆。
- 当信号数据暂不可用时，系统必须清楚显示缺失状态，而不是留空误导用户。

#### FR-9: 展示事件的延续性与历史关联

用户可以判断当前 `热点事件` 是一次性噪音还是某个连续主线的一部分。实现 UJ-2、UJ-3。

**Consequences (testable):**
- 系统可将某个 `热点事件` 关联到一个 `主题页` 或历史相关事件列表。
- 用户能从当前事件跳转到相关 `主题页`。
- 如果没有足够证据建立历史关联，系统不能强行生成“相似历史”。

### 4.4 日报与主题页

**Description:** `日报` 和 `主题页` 用来承接“持续阅读”和“复盘”需求，把单个 `热点事件` 组织成更长时间尺度的消费形式。实现 UJ-3。

**Functional Requirements:**

#### FR-10: 生成结构化日报

用户可以查看一个交易日的结构化 `日报`。实现 UJ-3。

**Consequences (testable):**
- `日报` 至少包含当日重点 `热点事件` 清单和每条事件的简要结论。
- 用户可以从 `日报` 直接跳转到对应 `热点事件详情页`。
- `日报` 明确标注覆盖日期和生成时间。

#### FR-11: 提供主题页进行连续主线追踪

用户可以通过 `主题页` 跟踪某个热点主题的连续演化。实现 UJ-3。

**Consequences (testable):**
- 一个 `主题页` 可以承载多个时间不同的 `热点事件`。
- `主题页` 需要按时间顺序展示与主题相关的事件变化。
- 用户可以从 `主题页` 返回任一关联的 `热点事件详情页`。

### 4.5 搜索与关注列表

**Description:** V1 不做重社交，但必须让用户能快速回到自己关心的内容。`搜索` 和 `关注列表` 是轻量留存能力，而不是完整用户关系系统。实现 UJ-2、UJ-3。

**Functional Requirements:**

#### FR-12: 支持跨热点事件与主题页搜索

用户可以搜索 `热点事件`、`主题页` 或相关关键词。实现 UJ-2。

**Consequences (testable):**
- 搜索结果至少覆盖 `热点事件` 标题、解释摘要和 `主题页` 名称。
- 搜索结果按相关性与时间综合排序。
- 无结果时必须给出明确反馈。

#### FR-13: 支持收藏热点事件与主题页

用户可以将 `热点事件` 或 `主题页` 加入 `关注列表`。实现 UJ-2、UJ-3。

**Consequences (testable):**
- 用户可以从详情页和列表页执行收藏操作。
- `关注列表` 可以单独访问。
- 收藏状态在同一账号会话中保持一致。

### 4.6 运营复核与内容治理

**Description:** 如果没有 `运营复核`，V1 很容易因为错误归组、错误解释或来源缺失而失去可信度。这个功能服务内部运营，但它直接决定外部产品质量与合规性。

**Functional Requirements:**

#### FR-14: 支持运营复核热点事件归组与标题

内部运营人员可以对 `热点事件` 的归组、标题和标签做人工修正。

**Consequences (testable):**
- 运营人员可以合并、拆分或修正 `热点事件` 的标题和基础标签。
- 修正后，用户侧展示内容与运营结果保持一致。
- 系统记录一次修正的时间和操作者。

#### FR-15: 支持运营复核解释与来源完整性

内部运营人员可以检查解释内容和 `证据时间线` 的完整性。

**Consequences (testable):**
- 运营人员可以标记某个 `热点事件` 为待复核、已复核或需下线。
- 当 `证据源` 缺失或解释明显不可靠时，运营人员可以阻止该事件进入公开展示。
- 被 AI 生成的内容在运营侧也必须可识别。

## 5. Cross-Cutting NFRs

- **NFR-1 Timeliness:** [ASSUMPTION: 对于进入 V1 重点范围的 `证据源`，从系统采集到出现在公开 `热点事件流` 的目标延迟应控制在 10 分钟以内。]
- **NFR-2 Traceability:** 所有公开展示的 `热点事件` 都必须能回溯到至少一个有效 `证据源`。
- **NFR-3 Transparency:** 所有 AI 生成的摘要、解释或衍生内容都必须做**显式 + 隐式元数据**标识（对齐《人工智能生成合成内容标识办法》2025.9.1 施行）；显式标识面向用户，隐式元数据标识（提供者名称/编码等）面向审计与监管。
- **NFR-4 Responsive Web:** V1 作为 Web 首发产品，必须在桌面端和移动端浏览器上可正常使用。
- **NFR-5 Graceful Degradation:** 当 `市场反应信号` 或部分外部数据不可用时，产品仍可展示 `热点事件` 和 `证据时间线`，并明确缺失状态。
- **NFR-6 Auditability:** 与 `运营复核` 相关的关键变更必须保留审计记录。
- **NFR-7 AI Provenance:** 所有 AI 生成内容（`AI 解读`/`AI 深读`/`趋势研判`/摘要/日报）每版必须保留 model id + prompt 版本 + 生成时间戳 + 留存期，审计可取（扩展 NFR-6 范围至 AI 生成 provenance）。

## 6. Non-Goals (Explicit)

- 不提供买入、卖出、仓位、目标价、收益承诺等投资建议。
- 不提供自动交易、组合跟单、策略订阅或信号推送。
- 不在 V1 构建重社区评论区、聊天广场或用户互动内容生态。
- 不在 V1 做机构终端级全量数据库、Excel 插件或专业 API 平台。
- 不在 V1 追求个性化推荐系统，而优先验证通用 `时间流` + AI 解释的价值。
- 不做脱离证据溯源的纯资讯搬运：`时间流` 每条动态必须可回溯到至少一个 `证据源`，`AI 解读` / `AI 深读` / `趋势研判` 不得编造无来源支撑的结论（对齐 NFR-2、NFR-3）。
- 不做无人工闸门的完全自动发布：`时间流` 条目与 AI 生成内容仍受 `运营复核` 抽检，不采用纯自动上线模式。
- 不弱化差异化护城河：`证据时间线`、`市场反应信号`、`运营复核` 三件套构成 aguhot 与纯资讯聚合器的唯一差异，任何为提升时间流流畅度而弱化这三件套的取舍需显式记录为 Major 决策。

## 7. MVP Scope

### 7.1 In Scope

- A 股公开信息驱动的 `时间流`（含同事件精选）
- `AI 解读`（列表卡钩子）
- AI 深读（事件级"为什么重要"）
- AI `趋势研判`（日报/主题页跨事件）
- `热点事件详情页`
- `证据时间线`
- `市场反应信号` 展示
- `日报`
- `主题页`
- 搜索（覆盖 `时间流` 条目、`热点事件`、`主题页`）
- `关注列表`
- `运营复核`（含对 AI 生成内容的抽检）

### 7.2 Out of Scope for MVP

- 实时推送订阅
  - 原因：先验证拉式消费场景，再决定是否做推式分发。
- 用户评论与观点内容生产
  - 原因：会显著放大审核与社区治理成本。
- 付费会员体系
  - 原因：商业化假设仍待验证，不应先于核心价值成立。
- 机构版协同工作台
  - 原因：与 V1 的轻量解释层定位不一致。
- 主题页 AI `趋势研判`
  - 原因：趋势研判是最高风险 AI 层（跨事件因果综合），MVP 仅落日报页（事件集有界、编造风险可控）；主题页研判待 §12 Q3 主题页生成方式定案且日报研判 SM-6 连续 2 周达标后启动（v1.1）。

## 8. Success Metrics

**Primary**

- **SM-1:** `时间流` 首页到 `热点事件详情页` 的点击率 >= 35%。验证 FR-1、FR-4。
- **SM-2:** `热点事件详情页` 平均停留时长 >= 2 分钟。验证 FR-4、FR-5、FR-8、新增 AI 深读。
- **SM-3:** `证据源` 追溯覆盖率 = 100%。验证 FR-5、FR-15。
- **SM-7:** `时间流` 每条动态的 `AI 解读` 覆盖率 >= 95%（剩余允许缺失态，不得留空）。验证 FR-1、NFR-3。

**Secondary**

- **SM-4:** 登录用户 7 日留存 >= 20%。验证 FR-10、FR-11、FR-13。
- **SM-5:** `日报` 打开率 >= 25%（含 AI `趋势研判` 区块）。验证 FR-10、新增趋势研判。
- **SM-6:** 运营复核后被判定为”解释明显误导”的公开条目（含AI 解读/深读/研判）占比 < 10%。验证 FR-15、NFR-3。
- **SM-8:** 深度阅读占比 >= 基线值（基线 = Epic 4 上线前最后一周冻结值，PM 前置；深度阅读 = 会话内触达 `证据时间线` 或 `AI 深读` 区块 / `时间流` 首页 DAU）。`时间流` 首页 DAU 中位会话时长 >= 5 分钟为 secondary。验证时间流未牺牲深度（对冲 SM-C1）。

**Counter-metrics (do not optimize)**

- **SM-C1:** 不以 `时间流` 页面浏览量/刷新次数最大化为目标，避免沦为纯资讯刷量。对冲 SM-1、SM-8。
- **SM-C2:** 不以 `时间流` 条目数量最大化为目标，避免同事件不折叠、噪音过多。对冲 SM-1、SM-7。
- **SM-C3:** 不以 AI 文案长度最大化为目标，`AI 解读` 保持一句话、`深读`/`研判` 有上限。对冲 SM-2、SM-7。

## 9. Constraints and Guardrails

### 9.1 Safety

- 产品不得用任何页面、按钮、标签或订阅文案暗示投资建议属性。
- 当事件仍存在高不确定性时，系统必须允许显示“未确认”状态，而不是强行给出确定结论。

### 9.2 Privacy

- [ASSUMPTION: V1 可以在不强制登录的前提下让大部分用户完成浏览。]
- 默认最小化采集个人信息，优先支持匿名浏览或轻量账户能力。
- 如果后续加入个性化能力，必须支持透明说明与关闭机制。

### 9.3 Cost

- [ASSUMPTION: V1 不依赖昂贵的全量机构级数据终端采购，而优先用可承受的数据源组合验证产品价值。]
- `运营复核` 的流程设计必须支持小团队运行，不能默认依赖大规模人工编辑部。

## 10. Compliance and Regulatory

- 产品边界定位为金融信息服务中的信息聚合与事件解释，不进入证券投资咨询业务边界。`时间流` 资讯聚合 + AI 编辑点评形态很可能落入金融信息服务范畴，须走正式合规复核（§12 Q9）。
- 所有公开展示的 `时间流` 条目与 `热点事件` 必须保留 `证据源`、来源名称、时间和原始链接。
- 所有 AI 生成内容（`AI 解读` / `AI 深读` / `趋势研判` / 摘要 / 日报）必须做显式 + 隐式元数据标识（NFR-3）并保留 model id + prompt 版本 + 时间戳（NFR-7），允许 `运营复核` 干预。
- 产品不得使用直接诱导型措辞；`AI 解读` 生成须受措辞黑名单约束，黑名单覆盖六类：动作类（买入/卖出/增持/减持/建仓/清仓/加仓/减仓）、收益预测类（必涨/必跌/稳赚/暴涨/翻倍/连板）、操纵框架类（主力/庄家/拉升/出货/洗盘）、推荐强度类（强烈推荐/首推/首选/重点推荐）、时点建议类（抄底/逃顶/止损位/目标价区间/仓位）、过度确定类（必将/确定上涨/一定）。黑名单以正向可枚举常量承载，违反即拒绝重试或落缺失态。
- AI `AI 解读` 视觉权重 <= 事实摘要，不得在卡片上比事实标题/摘要更突出（等同编辑背书/投资指向风险）。
- [待复核] `时间流` + AI 编辑点评触发三类合规面，GA 前须出具外部律所书面意见：(1)《互联网信息服务算法推荐管理规定》排序精选类算法备案（`AI 解读` + 置顶/精选属排序精选类，财经+舆论属性须向网信部门备案）；(2)《人工智能生成合成内容标识办法》显式+隐式标识（NFR-3）；(3) 金融信息服务管理规定报备。复核结论出具前，`时间流` 不对外公开访问、不接入付费/推广位、不宣传为”推荐”或”精选投资资讯”（dev 期间以”内测不公开”形态运行）。
- 若未来新增个性化推荐，应同步补足算法透明、标签管理和关闭机制。
- 若未来新增面向机构的专业终端或 API，需要重新评估金融信息服务报备边界。

## 12. Open Questions

1. V1 的首批 `证据源` 组合具体包括哪些来源，优先级如何排序。
2. `市场反应信号` 在 V1 中具体使用哪些指标，哪些指标可稳定获得。
3. `主题页` 是完全自动生成，还是需要 `运营复核` 参与。
4. `关注列表` 是否需要登录态，还是先支持本地化保存。
5. [已收口] `时间流` 默认按交易日分组，组内按分钟级时间倒序；交易日划分为盘前 / 盘中 / 盘后三段标签，用户可按段筛选。自然日作为非交易日 fallback 分组。
6. [已收口] `时间流` 的"同事件精选"折叠阈值 = 2：同一 `HotEvent` 关联 ≥2 条 `EvidenceSource` 即折叠为单条"同事件精选"条目，展开列各来源；`HotEvent` 仅 1 条证据源或未聚类的单源快讯独立成条，不挂"同事件精选"标签。阈值以 `TIMELINE_FOLD_THRESHOLD` 配置项承载（默认 2），运营后台可调，不写死。
7. [已收口] AI 生成时机与可见性：`AI 解读` / `AI 深读` 在 event-assembly 聚类完成、候选 `HotEvent` 生成后即生成（候选阶段生成，供运营复核时一并审），公开侧仅 `publication_status = published` 后可见（AD-6 闸门）；不在证据入库时生成。`趋势研判` 在日报生成 job / 主题页刷新时生成（依赖多个已发布 `HotEvent`），随日报/主题页发布。候选阶段生成的 AI 内容进运营复核台作为复核素材，但不进公开 `published_timeline`。
8. [已收口] 分层闸门：事实层时间流条目经 source-ingest + event-assembly + NFR-2（100% 证据源覆盖）后自动发布，受发布后运营抽检（不逐条发布前复核，保 NFR-1 10 分钟延迟）；AI 生成内容（`AI 解读`/`AI 深读`/`趋势研判`）随事件 `published` 自动上线但纳入强制抽检（SM-6 < 10%），其中高风险子集（个股-facing、研报/公告类、含收益/目标暗示）需发布前运营签核。”低风险”由正向白名单定义，模糊项默认拒绝放行。满足 §6 红线（非纯自动上线）。
9. [已收口] 假设三类合规义务均触发（算法推荐备案 + AI 标识办法 + 金融信息服务管理），阻塞 GA 与商业化，不阻塞 dev；Epic 4/5 dev 期间以”内测不公开”形态运行，PM 同步发起外部律所书面意见（2 周窗口）。算法备案与隐式标识须进 Epic 5 AC，不得拖到商业化才补。
